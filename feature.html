World Interaction System

To interact with the world, and that literally includes almost everything - from kindling a light, opening a chest, to picking up an object or talking to an NPC - I have a single all-purpose interaction interface that each interactable object can implement as it sees fit. To complement the interaction interface, I have an interaction info widget and an interaction input key. With the interaction key the player can, for example, start a conversation with an NPC.

The environment scanner tries to detect the environment for interactable objects. As soon as it finds one, the info widget pops up, and fills the info slots with the information that the interactable object provides: the correct icon, the name, a description, and a prompt of what to do next. How good this system performs, depends on the implementation of the environment scanner. My implementation is slightly different than Tom Looman's version. First I use an overlap sphere. The overlap sphere is limited to a custom interaction channel. Then I use dot product filter to filter out the interactable object that is right in front of me and I also take the distance into account to some degree.

Combat System

Achieving good, fluid combat is tough. I'd say the most difficult part is not even a programming problem, but having an eye to separate the good animations from the bad ones. In my experience, there are three key aspects to look out for: a short recovery time, the animation should be somewhat symmetrical or give a sense of completeness, and third, too much root motion can be problematic. Let me give you an example for the second point, as I believe this is the least intuitive to see. I found that animations firing off two or three punches in one go often work better than single punch animations. The reason is that when you punch a couple of times in one go, they create a rhythmic pattern, that you don't get with a single punch. This is a trick that you see, for example, in Diablo.

Other than picking the right animation from the get-go, what we can do to mitigate long recovery times, is to implement animation canceling. The simplest way to do this is to use an animation notify state that opens and closes the window when animation canceling is allowed. In my system, I use at least two windows: one to connect attack moves into each other, and a second window to cancel into locomotion animation. The latter opens up at a later frame.
